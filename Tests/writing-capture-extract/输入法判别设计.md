# 输入法判别设计 —— 用 `input_source` 区分拼音 / 英文

> 目的:在 keystroke 采集时记录**当时所用的输入法**,把"这串拉丁字母是中文拼音还是英文"
> 的判别,从"靠词典/启发式猜"升级为"直接读模式"。本文只讲**判别逻辑**;采集层落地见末节。

---

## 1. 问题:拉丁字母的二义性

`keystroke_log` 记录的是物理击键的 Unicode 字符。中文输入法下打"妈的",键盘上敲的是
`m-a-d-e` + 选字数字 —— **keystroke 里存的是拉丁字母 `made`,不是汉字"妈的"**
(`Schema.swift:489-491` 早已注明:"拼音字母 + 选词数字键,拿不到合成的汉字;
中文 char 永远是 latin 字母")。

于是同一串 `made`:

| 场景             | 应得结果                      |
| ---------------- | ----------------------------- |
| 中文输入法下打的 | "妈的"的拼音 → **解码成中文** |
| 英文键盘下打的   | 英文单词 made → **留字面**    |

光看字符流分不出来。而且这类"既是合法拼音又是英文词"的**双关词**
(made/make/let/wait/ming/an/bei/men/long/dang/shang/hai...)**无法穷举硬编码**,
词典法(现有 `_is_eng_tail` 查 rime 英文词典)对它们必漏 —— rime 把 `made` 优先解成"妈的"。

> 实测验证(2026-06-13):`made→妈的`、`make→马克`、`let→乐天`、`wait→外套` 全被 rime
> 优先判中文。靠加白名单穷举双关词 = 死路。

---

## 2. 核心信号:`input_source`(权威模式)

新增 `keystroke_log.input_source` 字段,记录每次击键时的**当前输入源 ID**。
它直接告诉你"这个字母当时是在什么输入法下打的":

| `input_source` 形态                              | 类别     | 含义                     |
| ------------------------------------------------ | -------- | ------------------------ |
| `com.apple.keylayout.US` / `.ABC` / `.British` … | 键盘布局 | **英文字面模式**         |
| `im.rime.inputmethod.Squirrel`(鼠须管)           | 输入法   | 中文拼音                 |
| `com.apple.inputmethod.SCIM.ITABC`(系统简体拼音) | 输入法   | 中文拼音                 |
| `com.sogou.inputmethod.*` / 微信输入法 / 百度 …  | 输入法   | 中文拼音                 |
| `com.apple.inputmethod.Kotoeri.*`(日文) / 韩文 … | 输入法   | 其他语言(不在本判别范围) |

**归一判据**(pipeline 侧,对存下的 ID 字符串做):

- ID 含 `keylayout` → **英文字面**
- ID 含 `inputmethod` 且属已知中文 IME → **中文拼音**
- 其他语言 IME → 各自处理(留字面 / 各自解码),不进中英判别

> 更稳的程序化判法是 macOS 的 `kTISPropertyInputSourceType`
> (`kTISTypeKeyboardLayout` vs `kTISTypeKeyboardInputMode`),但存原始 ID 字符串
> 可读、可调试,pipeline 用子串归一即可。**存 ID,分类在 pipeline。**

从"猜模式"变成"读模式" —— 这是质变。

---

## 3. 判别决策树(分层)

一段连续拉丁字母 → 怎么决定**字面英文** or **拼音解码**:

```
拉丁段(连续字母,可含夹杂的数字/标点)
│
├─ 这段击键有 input_source 吗?
│   │
│   ├─ 英文键盘(keylayout) ──────────────→ 【字面英文】整段保留,不解码
│   │                                        ✓ zhangye5 / bug / ok / made(英文)
│   │
│   └─ 中文输入法(inputmethod)
│        └─ 进 IMEStateMachine(逐键重放组合区状态):
│             ├─ 字母            → 压入拼音组合区
│             ├─ 数字 + 组合区非空 → 【选字 → 中文】(确认了候选)
│             │                       ✓ men2→门 / pai3→排 / made+选字→妈的
│             ├─ 数字 + 组合区空   → 【字面数字】(没有待选拼音)
│             │                       ✓ 门5 的 5 / 排32 的 32
│             ├─ 空格/标点/回车     → 清组合区(提交 or 残渣)
│             └─ 字母进了组合区但全程无选字就发送 → 拼音残渣
│                  └─ 弱信号:中文 IME 内却零选字 ⇒ 疑似"输入法内 shift 英文"
│                       → 交 OCR / 上下文 兜底
│
└─ 没有 input_source(历史数据 / 偶发拿不到)
     └─ fallback 到现有链:
        选字数字信号 → IMEStateMachine → 词典(_is_eng_tail) → OCR
```

**两个正交的判别维度,别混**:

1. **中文 vs 英文**(整段模式):`input_source` 一锤定音(除"输入法内 shift 英文"洞)。
2. **数字是字面还是选字**(中文模式内):靠**组合区状态**(按数字时组合区空=字面、满=选字),
   `input_source` 帮不了这一维 —— 它俩**互补**,不是替代。

## 4. 关键案例走一遍

| 消息                   | 输入过程            | `input_source`    | 判别结果                                       |
| ---------------------- | ------------------- | ----------------- | ---------------------------------------------- |
| `微信号叫zhangye5`     | 切 US 键盘打 ID     | `keylayout.US`    | `zhangye5` 整段**字面** ✓(不再误解码)          |
| `我们在门5`            | 拼音打"门"→切字面 5 | `inputmethod`(门) | "门"选字解码;`5` 组合区空=**字面数字** ✓       |
| `这玩意现在全球排32`   | 拼音 + 末尾字面 32  | `inputmethod`     | 排=选字;`32` 组合区空=**字面** ✓(修了 ev1081)  |
| `ji d`(→记得)          | 拼音简拼 + 选字 1   | `inputmethod`     | 有选字 → **解码"记得"** ✓                      |
| `Nice! lemme check en` | 英文键盘            | `keylayout`       | 整段**字面英文** ✓(修了 ev1013,不再"了么么车") |
| 打"妈的"(`made`)       | 拼音 made + 选字    | `inputmethod`     | 有选字 → **"妈的"** ✓                          |

**双关词 `made` 的关键**:英文键盘 = 字面 `made`;拼音输入法 + 选字 = "妈的"
—— `input_source` **直接把两者分开,根本不用判别 `made` 这个词本身**。这就是
"无需穷举双关词"的根本原因:不判词,判用户当时用的输入法。

## 5. 边界与 fallback(诚实)

1. **历史数据无字段**。5/27–6/5 的 keystroke 没有 `input_source`,老数据继续走现有链
   (AX commit 流 / 选字信号 / 状态机 / 词典)。新字段**只对之后采集的输入生效**。

2. **输入法内 shift 英文 —— 唯一拿不到的洞**。`TISCopyCurrentKeyboardInputSource`
   拿到的是**输入源**(鼠须管 vs US 键盘)。你在鼠须管**内部**按 shift 临时打英文,
   `input_source` 仍是"鼠须管",API 看不到它内部的英文子状态。
   - **覆盖**:系统级输入源切换(切到 US/ABC 键盘打英文)✓
   - **不覆盖**:中文输入法内 shift 打英文 ✗
   - **缓解**:二级弱信号 ——「中文 IME 模式 + 这段全程无选字数字」⇒ 疑似内部英文,
     交 OCR(屏幕渲染事实)/ 上下文 兜底。这是判别链最后那点极窄残留。

3. **字面数字 vs 选字数字不归 `input_source` 管**。它只定"中/英模式";数字是不是选字,
   由组合区状态机判(按数字时组合区空/满)。两者必须都接上才完整。

## 6. 与现有判别链的整合

`input_source` **不取代**现有逻辑,而是把判别链的**第一刀从"猜"变成"知"**:

```
旧链: 词典(_is_eng_tail,双关词漏)→ 选字信号 → 状态机 → OCR → ?
新链: input_source(权威中/英模式)
        → IMEStateMachine(中文模式内:字面数字 / 选字)
        → OCR(残渣兜底)
        → LLM(极窄:输入法内 shift 英文 + 双关 + 无 OCR,零写权判别)
```

- **真相优先级**:AX edit_log commit 流(IME 实际上屏什么) > `input_source`(当时什么模式)
  > 击键状态机(从字符流逆推) > OCR > LLM。正常采集 AX commit 已对;`input_source`
  > 专治"AX 竞速留下拉丁残渣、需要从击键重建"的场景。
- **LLM 的正确位置**:只在最后一层兜"中文 IME 内 shift 英文且无 OCR"的双关残渣,
  零写权(写权恒在确定性链路的 guard 手里),不碰"14B 拿删/改权必伤真内容"的铁律。

---

## 7. 数据格式与采集落地

**字段**:`keystroke_log.input_source TEXT`(nullable;历史行为 NULL)。存**原始 ID 字符串**
(如 `com.apple.keylayout.US`),保留全部信息供调试;分类(中/英)在 pipeline 侧做。

**采集改动**(Swift 主项目,三处,均有先例):

1. **Schema migration**(仿 `v24` 加 modifiers):
   `ALTER TABLE keystroke_log ADD COLUMN input_source TEXT`(`DB/Schema.swift`)
2. **`KeystrokeEntry` struct** 加 `var inputSource: String?`(`Typing/KeystrokeCharLogger.swift:101`)
3. **`ingest()`** 在 callback 线程(已在同步读 `frontmostApplication`,同一处)调
   `TISCopyCurrentKeyboardInputSource` 取 `kTISPropertyInputSourceID`,写入字段
   (`Typing/KeystrokeCharLogger.swift:51-95`,需 `import Carbon`)。

**性能**:每键调 TIS 会建 CF 对象,有开销;`frontmostApplication` 也是每键读 cached 值、
实测稳,所以可先直接每键调。真嫌重 → 监听 `kTISNotifySelectedKeyboardInputSourceChanged`
通知缓存当前输入源,`ingest` 读缓存。

**注意**:改 Schema migration + CGEventTap 热路径;落地后需在 Xcode 重新 build,
新字段只对之后的输入生效。
