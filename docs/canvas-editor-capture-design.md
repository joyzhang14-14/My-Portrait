# Canvas 编辑器写作采集 —— 设计文档

> 状态:**草稿,讨论中**。不急,慢慢迭代。
> 解决的问题:Google Docs 这类 canvas 编辑器,typing observer 抓不到正文。

---

## 1. 背景

typing observer 靠 Accessibility (AX) 读输入框的值来抓打字。但一类 app
**把内容画在 `<canvas>` 上、DOM/AX 里没有文字**:

- Google Docs / Sheets / Slides
- Figma / FigJam
- 网页白板:Excalidraw / tldraw / Miro / Canva

用户**大部分写作在 Google Docs**,所以这条采集路必须解决。已确认走不通的:

- AX value —— 焦点元素是隐藏 IME 输入框,只有拼音碎片
- AX side-DOM —— 即使开屏幕阅读器支持,也只有菜单 chrome + 播报碎片,没正文
- Google Docs API —— 个人版要么每周重授权,要么走 Google 审核
- 服务账号 API —— 可行但要一次性 GCP 设置 + 文件夹共享,且没有过程信息

结论:**canvas 编辑器需要一套独立的、多源融合的采集路径。**

---

## 2. 总体思路:多源融合

没有任何单一来源能干净拿到「写了什么 + 怎么写的」。改成**四层来源各司其职,
LLM 按时间对齐融合**:

| 层 | 来源 | 抓什么 | 可靠度 | 何时生效 |
|----|------|--------|--------|----------|
| L1 | AX + splice 算法 | 内容 + 逐窗口编辑流水 | 高 | 普通 app(canvas 编辑器失效) |
| L2 | 屏幕 OCR | 内容(屏幕渲染的字) | 中高 | canvas 编辑器的**内容主源** |
| L3 | 原始击键字符 | 内容粗猜 / 交叉印证 | 低(中文是拼音) | canvas 编辑器 |
| L4 | 击键时间流 | 过程证据(节奏、停顿、退格量) | 高(纯时间信号) | canvas 编辑器 |

- 普通 app 走 L1,已有,不变。
- **canvas 编辑器走 L2 + L3 + L4,LLM 融合。**
- L2 给「写了什么」;L4 给「怎么写的」(两次 OCR 快照之间打了又删的 churn,
  纯 OCR 看不见,靠击键量 + 退格数补上);L3 是弱的内容旁证。

---

## 3. 采集机制

### 3.1 OCR(L2 —— 已有,需小改)

屏幕采集 + OCR 已经在跑,`frames` 表已有 OCR 文本 + 时间戳 + app(+ v17 起
浏览器 URL)。canvas 编辑器要补一个优化:

- **裁正文区再 OCR** —— Docs 窗口顶部工具栏、左侧栏的菜单(文件/编辑/查看…)
  会被 OCR 当噪音。OCR 前先裁到正文页矩形 → chrome 进不来。
- 正文区定位办法待定(见待讨论)。

### 3.2 带闸门的击键日志(L3 + L4 —— 新)

`KeystrokeLedger` 现在只存按键时间戳(给关联判据用)。扩展:

- **焦点在 canvas 编辑器时**(按 app + URL 判定),额外记录每次按键的:
  字符(`keyboardGetUnicodeString`)、时间戳、是否退格。
- 落一张新表 `keystroke_log`。
- **隐私闸门**:
  - 只在 canvas 编辑器里记,**不全局记**(全局记 = 键盘记录器)。
  - 沿用现有 typing 黑名单(密码管理器 / secure field / 终端一律不记)。
- 中文场景下字符流是拼音,所以 L3 只是弱旁证;L4(时间)才是它的主要价值。

### 3.3 LLM 融合 worker(新)

后台批处理(夜间 / 调度器触发,非实时):

- 取一段 canvas-编辑器 session 的 **OCR 帧文本(带 ts)+ 击键日志(带 ts)**。
- 喂给本地 LLM,按时间对齐,产出**结构化写作记录**:内容 + 过程描述
  (有没有重复打同一段、有没有删改、节奏)。
- 落进 `typing_events`(复用)或新表(见待讨论)。

---

## 4. 模型

- **阶段一:本地 Qwen3-4B**(MLX 跑,Apple Silicon 原生)。
  - 这是个「多源对齐 + 整理」任务,不是难推理 —— 4B 先试,不够再上 8B。
  - 后台 worker 跑,不要求实时,本地小模型完全够。
- 不用 Codex(代码模型,不对路),不用前沿大模型(杀鸡用牛刀 + 隐私)。
- 市面**没有专门做这个任务的现成模型** —— 这是 niche 自定义任务。

---

## 5. 训练数据积累 —— 为「阶段三专用模型」铺路

目标:以后能 LoRA 微调一个更小(Qwen3-1.7B / 0.6B)、专门干这一件事、
贴合用户自己写作习惯的模型。微调要**训练对**,所以从现在起就要把数据攒好。

### 5.1 一个训练对长什么样

```
输入  = 一段 session 的 OCR 帧文本(带 ts) + 击键日志(带 ts) + URL/app
输出  = LLM 融合出的写作记录(内容 + 过程描述)
```

### 5.2 因此要额外做的事

1. **原始输入必须保留,不能用完即弃。** 默认逻辑可能处理完就删原始 OCR/
   击键数据省空间 —— 那就没法回头当训练集。需要一个**保留期**策略
   (原始 OCR + 击键日志留 N 个月,见待讨论)。
2. **每条写作记录链回它的源** —— 记下时间窗 + 源 frame id 列表 + 击键日志
   区间。这样能精确取出 (输入, 输出) 对。
3. **存 LLM 的 prompt + 原始输出原文** —— 训练对要精确、可复现。
4. (可选)**用户校对标记** —— 用户若修正过某条写作记录,标为「已校对」,
   这是金标准训练数据,微调时优先 / 加权。

### 5.3 三阶段路线

- **阶段一(现在)**:Qwen3-4B + 精心写的 prompt 跑起来。同时所有原始数据 +
  prompt + 输出全部留存。
- **阶段二(攒数据,几周~几个月)**:数据自然积累。(可选)用更强的「老师」
  模型(本地 8B 或借 API)重新标注一遍,提高训练标签质量。
- **阶段三(微调专用模型)**:用攒下的对 LoRA 微调 Qwen3-1.7B/0.6B。
  工具:MLX-LM(可直接在 Mac 上微调)/ Unsloth / Llama-Factory。

---

## 6. 隐私

- 击键日志:只在 canvas 编辑器记 + 同步 typing 黑名单。范围被框死。
- 全部本地,不上传(与 app 既有定位一致)。

## 7. 待讨论(open questions)

按重要性排:

1. **session 怎么切分?** canvas 编辑器没有可靠的 AX focus 事件 —— 一段
   「写作 session」靠什么定边界?时间空隙 + URL 变化?
2. **OCR 正文区怎么定位裁剪?** AXWebArea 的 frame?固定偏移?其它?
3. **写作记录的输出 schema?** 复用 `typing_events`(text + edit_log)还是
   新表?过程描述放哪、什么结构?
4. **原始数据保留多久?** 训练要它留着,但占空间。留多久、留多少?
5. **worker 触发时机?** 夜间定时?还是积累到一定量触发?
6. **阶段二的「老师」模型** 用本地 8B 还是借一次 API 大模型把标签做准?
7. **L3 击键字符到底用不用喂 LLM?** 还是只用 L4 时间、L3 丢掉?
8. **跨 OCR 帧的去重 / 拼接** —— 长文档滚动时 OCR 只见可见区,怎么拼?
9. canvas 编辑器清单(哪些 app/URL 触发这条路)怎么维护?复用 / 扩展
   现有黑名单 UI?

---

## 8. 一句话

普通 app 走老路(AX + splice);**canvas 编辑器走新路:OCR 管内容 +
击键流管过程 + 本地 Qwen 融合**。现在起留存原始数据,为以后微调专用小模型
铺路。
